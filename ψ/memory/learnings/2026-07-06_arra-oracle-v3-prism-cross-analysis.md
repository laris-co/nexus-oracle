# Learned: arra-oracle-v3 prism-10 cross-analysis against Turso/Spice.ai

- **Market position, settled**: arra-oracle-v3 shares Turso's premise (embedded SQLite as the storage unit) but not its defining feature (no primary/replica sync protocol between sibling Oracles — cross-Oracle "federation" today is just a peer-discovery registry, not data replication). It shares zero DNA with Spice.ai's federation model. Not a competitor to either — don't position it against them in any GTM material.
- **The concurrency wall Turso exists to solve, arra-oracle-v3 already hits and papers over**: `bunfig.toml` documents "parallel test files sharing oracle.db cause disk I/O error"; lock hits get mapped straight to an HTTP 503 rather than resolved. This is "resilience theater around an unsolved problem," per the Skeptic lens — worth fixing via a write-serialization queue before ever considering a storage-engine swap.
- **Two real, previously-unknown bugs surfaced with file:line evidence**: (1) bare `bun test` silently fans out into 5 nested `agents/*` git-worktree copies, multiplying test execution (6 files/42 tests from 1 target — reproduced, not theoretical); (2) `sqlite-vec.ts`'s hybrid search over-fetches `limit*3` then filters in JS, silently under-returning results under selective metadata filters — the same "missing fast-path" class of bug an independent benchmark caught in Turso itself.
- **Security posture gap named explicitly**: both Turso (SOC2 Type II, Doyensec pen-test) and Spice.ai (SOC2 Type II, Prescient Assurance) hold external attestations arra-oracle-v3 has none of — and arra's own plugin sandbox docstring already admits isolation is "error containment only, not a security boundary." Decision: disclose this explicitly in user docs now rather than pretend, defer a real pen-test until after cheaper fixes land.
- **Cheapest win identified**: Spice.ai ships one dated, milestone-linked `docs/ROADMAP.md`; arra-oracle-v3 has none across 60+ docs. Flagged as a ~30-minute, zero-risk import — this is the kind of asymmetric-value finding cross-project analysis is good at catching that single-project review would miss.
- A sprint backlog (8 hypothesis-driven tasks) was drafted and briefly created via TaskCreate, then deleted the same session at Nat's request — this round was scoped as a knowledge report only, no code changes. The full backlog (hypothesis/action/verify/expected per task) still lives in `ψ/writing/2026-07-06/arra-oracle-v3-sprint-plan.md` for whenever implementation actually starts; re-verify findings before acting since context may have shifted.

Why this matters: this is a repeatable pattern — deep-research a peer/competitor project, then run a 10-lens prism specifically asking "what does this comparison reveal about our own code" rather than treating research and self-review as separate exercises.
