# Learned: the concrete mechanics of connecting Spice.ai to arra-oracle-v3 via MCP

- **"spicechat" is not a real Spice.ai product** — zero GitHub hits. It's shorthand for either the `spice chat` CLI subcommand or the `/v1/chat/completions` HTTP API, both driven by `spicepod.yaml`'s `models:`/`tools:` sections.
- **The connection is structurally possible today, with one hard constraint and one legal workaround.** Spice's MCP-client stdio spawn (`crates/runtime-tools/src/mcp/catalog.rs:185-197`, `tokio::process::Command`) never calls `.current_dir()` — the spawned process inherits Spice's own cwd, and there's no `cwd:` field on `spicepod.yaml`'s `Tool` struct (would hard-fail YAML parsing, `#[serde(deny_unknown_fields)]`). The `command` string is checked against a `DANGEROUS_PATH_GLOB_SET` that explicitly **rejects absolute paths** (proven by their own unit test) — but `params.mcp_args` is only length/count-checked, not path-checked. Workaround: `from: mcp:bun` (bare command) + `params.mcp_args: /absolute/path/to/arra-oracle-v3/bin/mcp.ts`.
- **arra-oracle-v3's side is unusually cooperative for this**: stdio-only (confirmed — zero `StreamableHTTPServerTransport`/`SSEServerTransport`/`rmcp` hits repo-wide), **no auth of any kind** in default embedded mode (constructor performs zero credential checks), and `bin/mcp.ts`'s `launcherRoot()` derives repo root from `import.meta.url` rather than cwd — meaning it doesn't care that Spice can't set its working directory. These three facts together are what make the workaround viable at all.
- **Auto-discovery is unconditional**: `McpToolCatalog::list_tools` pages through `tools/list` and wraps every tool returned with zero filtering. arra-oracle-v3 exposes exactly 30 MCP tools (confirmed full list via `src/tools/mcp-manifest.ts`) — connecting would expose ALL of them to the Spice model at once, prefixed `arra_oracle__oracle_*` (Spice's `encode_tool_name` scheme, since MCP catalogs are never in the `default_catalog_names` allowlist).
- **This has never been done, by anyone, for any custom (non-published-package) stdio MCP server** — every real Spice.ai example (docs, cookbook, GitHub issues) connects to `npx`/`docker`/`uvx`-launched published packages (Google Maps, GitHub, filesystem, Atlassian). The `bun <script.ts>` pattern is derived purely from reading Rust source, not from any demonstrated recipe. Treat as plausible-by-code-reading, not proven.
- **Why this matters** — it upgrades [[project_arra-config-federation-key-insight]] from "Spice's pattern looks appealing" to "the specific mechanism that would make it work with our actual codebase is now traced end-to-end, file:line, on both sides." The missing piece is entirely on arra-oracle-v3's side: it has no equivalent of `from: mcp:<cmd>` to auto-import an external MCP server's tools into its own registry the way Spice can import arra-oracle-v3's.
