# Learned: live-testing Spice.ai's MCP federation surfaced a real PHI logging risk

- **Spice's stdio MCP client is a genuine host, not a bypassable proxy** — confirmed it spawns/holds the child process persistently and never reimplements tool logic, so a downstream fail-closed mask (youngdo-mcp's PHI gate) stays structurally in the critical path. Good news for anyone federating a sensitive tool through Spice.
- **Escalated finding, confirmed empirically (rounds 3-4)**: it's not "logs full output by default" — it's **"cannot be turned off at all via any documented mechanism, while using the standard `spice run` workflow."** Setting `runtime.task_history.captured_output: none` in spicepod.yaml, or an explicit `--set-runtime task_history.captured_output=none` CLI override, does NOT work — verified live by querying the actual `captured_output` column via `spice sql` after setting it both ways; full output was still present both times. Root cause: `spice run`'s CLI wrapper (`bin/spice/src/context.rs`) appends its own `--set-runtime task_history.captured_output=truncated` AFTER any user-supplied args — argv-order last-write-wins means the CLI's hardcoded default always silently wins, no error, no warning. The only thing that respects `none`: bypassing `spice run` entirely and invoking raw `spiced` directly. `captured_output` itself IS a documented spicepod option (v0.18.1-beta release notes) — what's undocumented is that `spice run` silently overrides it regardless of user config. This is a materially more serious bug class (silently-ignored security config) than "undocumented default."
- Also confirmed (round 3): the value returned to the tool caller is a pure untransformed passthrough on both call sites (`tool.rs` native-LLM path, `server.rs` MCP-gateway path) — `task_history` logging is a side channel, never touches what's actually returned.
- **A second independent egress path exists**: the same tool registry that feeds MCP federation also feeds Spice's native LLM chat loop (`tool_use.rs`) — a federated tool's (possibly-sensitive) output can reach whatever external LLM provider is configured, entirely separately from the MCP-transport-masking question.
- **Clean architectural split recommended for hybrid static/live data**: use Spice's native `duckdb` connector directly for data that's *already pseudonymized at rest* (no masking boundary to cross), but keep anything computed live with masking logic (youngdo's 4 dashboard tools: kpi/timeseries/breakdown/run_metric) behind MCP federation into the original gated process — don't try to reimplement live-masked logic as a native Spice connector.
- **Zero schema-drift risk, confirmed by tracing the actual call chain**: `McpToolCatalog::{all,all_definitions}` → `Tooling::tools()` has no caching anywhere — Spice re-fetches `tools/list` live on every resolution, never a pod-load-time snapshot. Measured cost is negligible (youngdo-mcp's own cold-boot round trip: 0.121s).
- **A real config bug was caught and fixed live, not just documented**: `mcp_args` is whitespace-split, not JSON — contradicts what the spicepod test fixture format implies at a glance. spiceai oracle built and tested a real working `spicepod.yaml` entry for a bun-run TS stdio server against an actual `npx` MCP server before handing it to youngdo-mcp, upgrading this from "plausible by source-reading" (like the arra-oracle-v3 stdio workaround) to "proven."

Why this matters: every other finding in this research arc (Turso adoption safety, arra-oracle-v3 MCP feasibility, stdio-vs-Streamable-HTTP taxonomy) was careful source-code inference, explicitly flagged as untested. This is the first place live testing actually happened, and it caught a real risk (default full-output logging) that reading the code alone missed — a concrete argument for live-testing any future MCP federation involving sensitive data before trusting a source-only trace.
