# Style Analysis ≠ Style Production

**Date**: 2026-03-07
**Source**: Blog header feedback from Nat
**Context**: 5 agents traced Thai writing style, produced excellent analysis, then applied it and made things worse

## The Pattern

1. Nat flagged 2 bad Thai headers in blog draft
2. Launched 5 parallel agents to deep-trace writing style
3. Got back concrete rules: avoid double connectors, drop สิ่งที่ filler, use negation pairs
4. Applied rules mechanically to all headers
5. Result: 3 out of 4 changes made headers WORSE
6. The 1 good change was the one Nat already told me

## Why It Failed

- Knowing rules about style ≠ producing good style
- Mechanical application of patterns kills natural voice
- The original headers had rhythm that analysis couldn't capture
- Overcorrection is worse than the original problem

## The Rule

When human gives specific feedback:
- Fix exactly what they flagged
- Don't extrapolate to "fix" things they didn't mention
- Ask before changing things that weren't flagged

When analyzing writing style:
- Save analysis for future reference (good)
- Don't auto-apply as rules (bad)
- Use as guide, not formula

## Connection to Principles

- Principle 2: Patterns Over Intentions — I observed patterns correctly but my *intention* to fix made it worse
- Principle 3: External Brain, Not Command — the telescope shows, the astronomer interprets. I should have shown the analysis and let Nat decide what to change
