# Learned: how Spice.ai actually connects to AI, and where MCP fits

- **Two independent axes, one shared registry.** Spice *calls out* to LLMs via a `Chat` trait (`crates/llms/src/chat/mod.rs`) with 8+ providers (OpenAI, Azure, Anthropic, Google, Bedrock, xAI, Databricks, local mistral.rs) — exposed as an OpenAI-compatible `/v1/chat/completions` and used internally by NSQL (text-to-SQL: same `chat_request()` call, just a specialized prompt/retry loop). Separately, Spice *serves and consumes* MCP (`/v1/mcp`) as a tool-exchange gateway — `RuntimeServer` (MCP server) and `McpCatalogFactory` (MCP client, importing e.g. GitHub/Google Maps tools). Both paths read/write the same `Arc<RwLock<HashMap<String, Tooling>>>` registry, so a tool defined once is callable by Spice's own LLM (native function-calling via `ToolUsingChat`) AND by any external MCP client — "AI connection" and "MCP" are two doors into the same room.
- **Spice's own hosted LLM is explicitly rejected as a chat backend in the OSS runtime** (`ModelSource::SpiceAI` → `LlmError::UnsupportedTaskForModel`, `model/chat.rs:176`) — whatever "Spice Cloud LLM" is, it's proprietary/not in this codebase; here it's model-*lister* only.
- **MCP timeline**: landed v1.1.0 (2025-03-31), ~4 months after MCP's Nov 2024 launch — genuinely early, 6-9 months ahead of Snowflake/BigQuery/Databricks/MongoDB's managed MCP offerings. v2.0.0 (2026-06-05) removed the old dedicated SSE server (`/v1/mcp/sse`) entirely, replacing it with Streamable HTTP at a single `/v1/mcp` endpoint, plus native auth + tool-tracing. MCP primitives used: **tools only** — zero use of `resources`/`prompts` anywhere in the repo.
- **Recursive federation is real**: an integration test proves one Spice instance configured as an MCP client of a second Spice instance re-exposes the second instance's tools (as `mcp_from_spiced__*`) through its own MCP server to a third client — the gateway/proxy angle chains, not just single-hops.
- **The tool-naming bug is the single best piece of evidence that MCP was bolted on, not co-designed.** Catalog-qualified tool names (`{catalog}/{tool}`) violated Claude/Cursor's `^[a-zA-Z0-9_-]{1,64}$` constraint — filed as #10887/#10894 (May 2025), fixed in PR #11629 / commit `41868398d` (merged 2026-07-05, the day before this research; verified live in the `mcp_spiced_list.snap` integration snapshot). But the *identical* constraint had already been fixed once, a year earlier, in PR #5116 (March 2025) — in a different code path (`model/tool_use.rs`, the internal-LLM-facing path) that was never unified with the MCP-server-facing path, which used its own unsanitized `format!()`. Four call sites had each independently reimplemented the same broken logic before #11629 finally consolidated them into a shared `crates/tools/src/naming.rs` module (`__` separator, escaped for lossless round-trip).
- **Spice's actual differentiator by mid-2026 isn't having an MCP server** (table stakes now across Postgres/Snowflake/BigQuery/MongoDB/Databricks) — it's federation: Spice proxies *other* MCP servers (GitHub, Google Maps) behind one endpoint alongside its own SQL tools, unlike single-source competitors. Confirmed structurally via the official cookbook demo, not just marketing copy.
- Independent commentary is thin (one educator tutorial reshared without comment by a co-founder, one marketplace listing, zero HN/Reddit/dev.to threads) — the "community sentiment" signal for this feature is really the GitHub issue tracker, worth noting if any future report cites broader buzz.
