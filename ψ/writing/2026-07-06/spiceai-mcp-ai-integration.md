# How Spice.ai Connects to AI — and What MCP Actually Does There

*Deep-dive via 4 parallel Sonnet agents (source-code read + web/GitHub research), 2026-07-06. Reference report only — no code changes, per standing "report first" instruction.*

## The short answer

Spice.ai connects to AI on **two independent axes that happen to share one registry**:

1. **Spice calls out to LLMs** — a `Chat` trait (`crates/llms/src/chat/mod.rs`) with providers for OpenAI, Azure, Anthropic, Google, Bedrock, xAI, Databricks, and local models (mistral.rs). Exposed as an OpenAI-compatible `/v1/chat/completions` and `/v1/responses` HTTP API, and used internally by NSQL (text-to-SQL) and by tool-calling.
2. **Spice serves and consumes MCP** — a completely separate wire protocol (`/v1/mcp`, JSON-RPC/Streamable HTTP) for *tool* exchange, not LLM exchange. Spice can be an MCP **server** (exposing its own SQL/data tools to Claude Desktop, Cursor, VS Code) and an MCP **client** (importing a third-party MCP server's tools, e.g. GitHub, Google Maps, into itself).

The reason these look entangled: both paths funnel into the **same tool registry** (`Arc<RwLock<HashMap<String, Tooling>>>` on the runtime). A tool defined once in `spicepod.yaml` is simultaneously callable by Spice's own configured LLM (via native function-calling) *and* by any external MCP client hitting `/v1/mcp`. So "Spice + AI" and "Spice + MCP" are two doors into the same room, not two different rooms.

## 1. The LLM layer (Spice calling out to models)

- Core abstraction: `pub trait Chat` (`crates/llms/src/chat/mod.rs:574`) — `run()` is the minimal method; `chat_request()`/`chat_stream()` give full OpenAI-shaped fidelity for providers that implement it directly (OpenAI, Anthropic, Google, Bedrock, xAI all override; others degrade through a default flattening path).
- Providers, one dir each under `crates/llms/src/<provider>/`: OpenAI, Azure (reuses OpenAI client), Anthropic, Google/Gemini, AWS Bedrock (+ Guardrails), xAI/Grok, Databricks (PAT or OAuth M2M/U2M), local via mistral.rs (HF/GGUF weights, even multi-node distributed "ring" inference).
- Notably: **Spice's own hosted "Spice Cloud" LLM is explicitly rejected as a chat backend in this OSS codebase** (`crates/runtime/src/model/chat.rs:176`, `LlmError::UnsupportedTaskForModel`) — it's a model-*lister* only here (`crates/llms/src/spiceai/mod.rs`). Whatever Spice.ai's hosted inference product is, it isn't in the open-source runtime.
- Dispatch: `try_to_chat_model()` → `construct_model()` matches spicepod's `ModelSource` enum and wraps the result in `ChatWrapper` (injects system prompt + per-request param overrides) — and, if `tools:`/`spice_tools:` params are set, wraps *again* in `ToolUsingChat` (see §3).
- HTTP surface: `/v1/chat/completions` (full OpenAI compat, streaming via SSE) and `/v1/responses` (OpenAI's newer Responses API, OpenAI/Azure-only for now).
- **NSQL is not a separate integration** — `/v1/nsql` builds a schema-context prompt, then calls `nql_model.chat_request(req)` (the identical `Chat::chat_request` used by chat completions), executes the returned SQL read-only against DataFusion, and retries with the error fed back into the next prompt on failure.
- Embeddings are a parallel `Embed` trait (OpenAI, Google, Bedrock, local candle-based, Model2Vec) exposed at `/v1/embeddings` **and** wired directly into DataFusion as SQL UDFs (`embed()`, `vector_search()`, `text_search()`) — so `SELECT * FROM vector_search(...)` is valid SQL that calls an LLM embedding model under the hood.

## 2. The MCP layer (Spice as gateway)

- **Landed in v1.1.0 (March 31, 2025)** — roughly 4 months after Anthropic shipped MCP (Nov 2024), making Spice.ai an early adopter — ahead of Snowflake, BigQuery, Databricks, and MongoDB's official MCP offerings (those arrived Oct 2025–Jan 2026).
- **v2.0.0 (June 5, 2026)** replaced the old dedicated SSE server (`/v1/mcp/sse`, PR #10491, commit `ed237ffbc`) with Streamable HTTP (`rmcp` v1.5.0) mounted at a single `/v1/mcp` endpoint — POST for JSON-RPC calls, GET for an SSE-formatted stream (this is the Streamable HTTP spec's own optional upgrade, not the legacy standalone SSE transport, which is gone), DELETE to terminate a session, continuity via an `Mcp-Session-Id` header. Also added native auth (`mcp_auth_token`, `mcp_headers`), task-history tracing of external MCP calls, `allowed_hosts` allowlisting (defaults to localhost/127.0.0.1/::1; requires `runtime.auth` configured or the endpoint 401s; 32MiB body cap), and a "searchable tool registry" (agents discover tools via semantic search instead of a giant system-prompt tool list).
- **Primitives: tools only** — `ServerCapabilities::builder().enable_tools().build()` (server.rs:87). Zero use of MCP's `resources` or `prompts` primitives anywhere in the repo. Built-in tool set (`crates/runtime-tools/src/options.rs:70-115`): `sql`, `search`, `list_datasets`, `table_schema`, `get_current_datetime`, `get_readiness`, `random_sample`, `sample_distinct_columns`, `top_n_sample`, `load_memory`, `store_memory`. Note "nsql" isn't a distinct MCP tool — it's a curated subset of these tools bundled under `SpiceToolsOptions::Nsql`.
- Server side: `RuntimeServer` (`crates/runtime-tools/src/mcp/server.rs`) implements `rmcp::ServerHandler`, constructed directly from the runtime's live tool registry (`RuntimeServer::new(Arc::clone(&runtime.tools))`, `routes.rs:466`). `call_tool` (server.rs:95-220) either executes a built-in directly — `sql`→`QueryEngine::execute_query` against DataFusion, `search`→`SearchEngine::search_with_cache` (hybrid vector+keyword), `list_datasets`→direct catalog introspection (no query execution) — or, if the tool is a proxy (`tool.as_mcp_proxy()`), transparently forwards the call upstream to whatever MCP server it came from. Both paths emit `task_history` tracing spans.
- Client side: `McpCatalogFactory`/`McpToolCatalog` (`crates/runtime-tools/src/mcp/{factory,catalog}.rs`) lets a spicepod `tools:` entry with `from: mcp:<id>` connect outward to a third-party MCP server — either **stdio** (spawns a child process, e.g. `docker run -i --rm mcp/fetch`) or **Streamable HTTP** (with its own `mcp_auth_token`/`mcp_headers`) — pulling its tools in as `Arc<dyn SpiceModelTool>`, indistinguishable from a built-in tool once registered. Has a 30s heartbeat/reconnect loop and hardening (path-traversal checks, arg caps, HTTPS required unless localhost).
- **Recursive proxying is real, not hypothetical**: an integration test proves a Spice instance can re-serve another Spice instance's MCP tools — i.e. Spice A configured as an MCP client of Spice B re-exposes B's tools (named `mcp_from_spiced__*`) through Spice A's own MCP server to a third client. The gateway/federation angle isn't limited to one hop.
- **Spice explicitly brands this a "gateway"** — https://spice.ai/feature/mcp-server-gateway pitches it as aggregating multiple distributed MCP servers behind one endpoint with centralized auth, fine-grained per-model/per-tool access control, and distributed tracing, aimed at regulated industries. This positioning is confirmed structurally, not just marketing: the official cookbook demo (`spiceai/cookbook/mcp-server`) proves `/v1/tools` returns Spice's own tools (`sql`, `list_datasets`, `search`, `store_memory`, etc.) **alongside** proxied GitHub MCP server tools (`github/create_issue`, `github/search_code`, ...) in one unified list.
- Real setup, straight from the cookbook: `claude mcp add --transport http spice http://localhost:8090/v1/mcp --header "X-API-KEY: foo"` — genuinely works as a live Claude Code/Desktop connection target today.

## 3. The unifying mechanism — one tool registry, two doors in, two doors out

- `Tooling` enum (`crates/runtime-tools/src/tooling.rs`): `Tool | FunctionTool | Catalog`. Registered factories include `"builtin"` (SQL, catalog browsing, readiness), `"memory"`, and (feature `mcp`) `"mcp"` → `McpCatalogFactory`.
- `ToolUsingChat` (`crates/runtime/src/model/tool_use.rs:53`) is a `Chat`-decorator: injects tool defs into the LLM request, executes matched tools locally, feeds results back, loops until an answer or a recursion limit (default 10). It pulls tools from the exact same `rt.tools` registry that `RuntimeServer` (the MCP server) reads from.
- Net effect: a tool defined once can be (a) called by Spice's own configured LLM via native function-calling, (b) called by an external MCP client via `/v1/mcp`, and (c) itself be a proxy to a *third-party* MCP server's tool. All three are the same object, entered/exited through different edges.
- **A real, recently-fixed bug exposes how bolted-on this unification still is**: tool names built as `{catalog}/{tool}` (e.g. `antv_charts/list_chart_types`) violate Claude/Cursor's `^[a-zA-Z0-9_-]{1,64}$` naming constraint. Filed as issues #10887/#10894 (May 2025), fixed in PR #11629 / commit `41868398d` (merged 2026-07-05, one day before this research) with a new shared `crates/tools/src/naming.rs` module (`__` separator, with `__`→`_-_` and `_-_`→`_--_` escaping for lossless round-trip) — verified live in the integration snapshot `mcp_spiced_list.snap`, which now shows names like `mcp_from_spiced__sql` with zero `/` characters. Notably **the identical constraint had already been fixed once before**, in PR #5116 (March 2025), in a *different* code path (`model/tool_use.rs`, the internal LLM-facing path) that was never unified with the MCP-server-facing path. Four separate call sites had each independently hand-rolled the same broken `format!("{catalog}/{tool}")` string before #11629 finally consolidated them. Not yet in a tagged release (latest is v2.0.1, predates the fix).
- Other MCP-surface rough edges found: task-history tracing didn't originally know about dynamically-named MCP proxy tools (#10995, #11055 — the latter's repro still showed the un-fixed `/`-separated names weeks after the bug was already filed); the built-in `sql` MCP tool hardcoded `read_only: true` regardless of the caller's actual API-key role (#11029), undermining agentic-write use cases via the MCP gateway specifically. All community-reported (mostly one non-maintainer, `felipemm`), not caught internally — consistent with MCP being a secondary surface layered onto a SQL/data-connector core, not co-designed with it from day one.

## 4. Market context

- Independent commentary is thin: one credible educator (Santiago Valdarrama, ~600k+ reach) published a hands-on MCP+Spice tutorial (reshared by co-founder Luke Kim without added commentary), one third-party MCP marketplace listing (cursor.directory). No Reddit/HN/dev.to threads specifically about Spice.ai's MCP integration were found — sentiment signal here is really the GitHub issue tracker, not public buzz.
- Spice.ai's actual differentiator by mid-2026 isn't "having an MCP server" (table stakes now — every major data platform has one) — it's the **federation/aggregation angle**: Postgres/Snowflake/BigQuery/MongoDB's MCP servers each expose only their own data as tools; Spice.ai's proxying of *other* MCP servers (GitHub, Google Maps, etc.) behind one endpoint alongside its own SQL layer is comparatively rare among data-platform MCP servers.

## Bottom line

Spice.ai's "AI connection" is really three things stacked: (1) a broad multi-provider `Chat`/`Embed` abstraction that lets *Spice's own* configured LLMs answer questions and generate SQL (NSQL), (2) an MCP gateway that lets *external* AI clients (Claude, Cursor, VS Code) call Spice's tools and, transitively, tools from other MCP servers Spice has imported, and (3) a shared tool registry that makes those two directions interchangeable. The recently-fixed tool-naming bug and its 2025→2026 recurrence in a parallel code path is good concrete evidence that (2) was retrofitted onto an architecture originally built around (1), rather than co-designed with it.
