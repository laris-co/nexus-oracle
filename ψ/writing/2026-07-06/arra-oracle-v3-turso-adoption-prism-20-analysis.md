## Precedent Analyst
**Finding**: `detectDbRuntime()` (src/db/factory.ts:43-48) is a hard **JS-runtime detector** (`isWorkerd` / `isBun` from std-env) — bun:sqlite and D1 are mutually exclusive by physical necessity (bun:sqlite doesn't exist in workerd; D1 only exists in workerd). The two `DbConnection` variants are structurally different (`BunDbConnection` carries a raw `sqlite: Database` + `StorageBackend`; `D1DbConnection` carries only a `d1` binding, no StorageBackend) — callers like handlers.ts's raw `sqlite.prepare()` FTS5 queries assume the bun:sqlite shape specifically.

Turso isn't a *runtime* — it's an alternative storage engine you could run on the very same Bun process, so it doesn't fit the isWorkerd/isBun if-chain at all; it needs a config/feature-flag axis, not a runtime-detection axis. The codebase already has the right template for that: `createVectorStore()`/`VectorDBType` (src/vector/factory.ts), which switches on config, not environment.

One good isolation fact: sqlite-vec opens its *own* separate bun:sqlite handle (src/vector/adapters/sqlite-vec.ts:49) independent of the core DB connection, so a Turso core-DB swap wouldn't automatically drag vectors along — but if Turso were ever asked to host vectors too, its `loadExtension()` incompatibility with real .so extensions (COMPAT.md) would block sqlite-vec outright.

**Recommendation**: Don't extend `detectDbRuntime()` — build a third `StorageBackend` implementation selected via a new config value (mirrors `VectorDBType`), feature-flagged off by default, pinned to v0.6.1 not HEAD.

## FTS5 Compatibility
**Finding**: FTS5 is a from-scratch C virtual-table module (custom tokenizers, MATCH query grammar, bm25 ranking, trigram/prefix indexes) baked into upstream SQLite's C codebase — not something a reimplementation gets "for free" by matching the file format. Turso is a ground-up Rust rewrite (not a libsqlite3 fork), and the context here already confirms Turso's extension model is explicitly non-compatible with native SQLite mechanisms (COMPAT.md: SQLite `.so`/`.dylib` loadable extensions "are not supported"). **Recommendation**: Don't do this yet. ## Rollback Safety
**Finding**: Today's core DB is a single flat file at `ORACLE_DB_PATH` (src/config.ts:49, resolved in src/db/create.ts:19-20) opened via `bun:sqlite`. Rollback today is trivial: stop the process, `cp oracle.db oracle.db.bak`, done — file-level atomicity, zero adapter code. No dual-write, no CDC, no export/import path exists between backends in the codebase today (export_jobs table at src/db/export-schema.ts is application-level export, unrelated to backend migration).

**Recommendation**: Don't do this yet as a live/prod swap. ## Concurrency Semantics Mismatch

**Finding**: Two concrete assumptions surface. ## Security Surface
**Finding**: Turso's attack surface would be genuinely new, not a repeat of an existing pattern. arra-oracle-v3 already loads native code today — sqlite-vec's `sqlite.loadExtension(sqliteVec.getLoadablePath())` (src/vector/adapters/sqlite-vec.ts:57, with fallback probe paths at :61) — so "native extension loading" itself isn't a novel category. But Turso's extension mechanism is a **different, hand-rolled C-ABI**, not standard SQLite .so/.dylib loading (COMPAT.md: SQLite loadable extensions explicitly unsupported). That means any security review done on sqlite-vec's loadExtension path doesn't transfer — it's a second, differently-shaped native-code trust boundary to audit from scratch. Additionally, Turso ships optional sync/network features (embedded replicas, libsql-server-style remote sync) that arra-oracle-v3 has zero use case for — unused capability is exactly the kind of surface that gets overlooked in review and exploited later (unauthenticated sync endpoints, credential handling for remote replicas). Compounding this: the version gap is severe — even Spice.ai's validated v0.5.1 is 2,737 commits behind HEAD, and arra-oracle-v3 has no SOC2/pen-test program of its own to catch what upstream review might miss. Adding an actively-mutating dependency's novel attack surface to a project with no independent security validation is stacking risk, not distributing it.

**Recommendation**: Don't do this yet. If piloted, use the existing `VectorStoreAdapter`/`createVectorStore()` pattern to isolate Turso behind a feature flag, disable sync/replica features entirely (config-locked, not just unused), and pin to v0.6.1 stable — never a v0.7.0-pre tag.

## Maintenance Burden / Exit Strategy
**Finding**: The bun:sqlite/D1 split (`src/db/factory.ts`, `detectDbRuntime()`) is *forced* — Workers literally cannot run bun:sqlite. Turso would be the first backend added by *choice*, not platform necessity, which changes the cost math: it's pure discretionary surface. Real costs: (1) 40 migrations (0000-0039) must stay valid across a 3rd SQL dialect; (2) FTS5 lives as raw `sqlite.prepare()` calls in `src/server/handlers.ts`, outside Drizzle — any Turso-equivalent keyword search needs its own bespoke raw-SQL path, doubling the un-abstracted surface; (3) Turso's non-standard extension ABI (confirmed: SQLite `.so`/`.dylib` extensions unsupported per COMPAT.md) means `sqlite-vec`'s `loadExtension()` trick cannot be reused — a Turso vector path would need a wholly separate adapter, not a shim; (4) pinning is mandatory (18 pre-release tags in <2 months) but pinning also means drifting behind fixes, mirroring Spice.ai's own 2,737-commit-stale reference build.

Good news: the existing `VectorStoreAdapter` pattern (6 interchangeable adapters, one factory switch) proves this codebase *can* isolate and delete a backend cleanly — IF Turso is built strictly behind that same seam and never leaks engine-specific SQL into `handlers.ts`.

**Recommendation**: Don't do this yet as a permanent 3rd backend. If experimenting, gate behind a feature flag, pin to stable v0.6.1 (not pre-release track), and confine all Turso code to one factory-selected adapter file with zero raw-SQL leakage into shared handlers — so removal is a one-file, one-branch deletion, not a migration rewrite.

## Minimalist / Smallest Safe Slice
**Finding**: The riskiest unknown isn't "can Turso run our app" — it's "does Turso even support the specific SQL surface we depend on." Two facts converge on this: (1) FTS5 is queried via raw SQL virtual tables (`oracle_fts`, `src/db/migrations/0017_fts5_bootstrap.sql`) since Drizzle can't model virtual tables, and it's the always-on fallback every degraded path lands on (`VECTOR_FALLBACK=fts5`); (2) Turso's extension mechanism is a hand-rolled C-ABI, explicitly not compatible with real SQLite `.so`/`.dylib` extensions (COMPAT.md). FTS5 itself is often compiled in rather than loaded as an extension, but whether Turso's Rust rewrite actually implements FTS5 virtual-table semantics (not just parses the SQL) is unverified and cheap to check. Nothing in package.json currently references libsql/turso — this is a zero-footprint question today.

**Recommendation — safe to try, and do it before anything else**: Write a single throwaway script outside the app (`scratch/turso-fts-probe.ts`, delete-when-done, no `src/` changes, no adapter interface). Point the Turso/libsql client at a local file, replay the 40 migration files verbatim including `0017_fts5_bootstrap.sql`, then run 2-3 real `MATCH` queries copied from `src/server/handlers.ts`. If FTS5 doesn't work or panics, stop — the fallback-path precondition fails and the rest of the investigation (adapter design, WAL bottleneck, SOC2) is moot. Don't build `VectorStoreAdapter`-style abstraction until this one script passes.
