## Archaeologist
**Finding**: The codebase shows layered accretion, not a single clean build. Evidence of succession: `bin/arra.ts` still dispatches on `argv[0]` for `arra-oracle-v2` alongside `-v3` — a v1→v2→v3 lineage where old binary names were never dropped. A completed framework swap is documented in-code ("Hono → Elysia migration is complete, no Hono left" per AGENTS.md/CLAUDE.md). Test tooling shows incomplete migration too: `vitest.config.ts` defines a coverage block but vitest isn't even a devDependency anymore — a stale leftover from "before the project standardized on Bun" — yet `src/indexer/__tests__/backup-lock.test.ts` still imports from `'vitest'` and silently passes via Bun's compat shim, meaning nobody finished the cleanup. Vector backend history shows a pivot: `ChromaMcpAdapter` (shells out via MCP, not a direct client) reads like the earliest, lowest-effort integration, later supplanted by native adapters (sqlite-vec, LanceDB, Qdrant, Cloudflare Vectorize) — but the switch was incomplete: `createVectorStore()`'s bare fallback is still `'lancedb'` while `generateDefaultConfig()` overrides to `sqlite-vec` specifically so "fresh installs... without a provider prompt" — two different defaults coexisting. `src/storage/migration-repair.ts` (211 lines, statement-by-statement drift detection) reads as reactive engineering built after real "duplicate column" pain from out-of-band DB changes. `src/plugins/sandbox.ts`'s docstring explicitly disclaims "process isolation, VM boundaries, or any security boundary" — a walked-back promise, now just exception containment.

**Recommendation**: Grep git blame/log on `src/vector/factory.ts` (lancedb fallback) vs `src/vector/config.ts` (sqlite-vec default) and `vitest.config.ts` deletion history to date these three unresolved-migration artifacts precisely — they're the highest-value archaeology targets for understanding *why* defaults diverged.

## Bug Hunter
**Finding**: Two concrete, evidence-backed defects stand out.

(1) **Test-runner path contamination is real and reproduced, not theoretical.** `bunfig.toml:1-8` sets `roots = ["src","tests"]`; because `agents/arra-codex-{backend,frontend,infra,research,test}/` are git worktrees containing full nested copies of `src/`+`tests/`, Bun treats any test path as a suffix match rather than exact path. Empirically confirmed: invoking `bun test --isolate src/indexer/__tests__/backup-lock.test.ts` with one explicit relative path ran the same-named file in all 5 nested worktrees too — 6 files, 42 tests total. `AGENTS.md:94-95` had to explicitly warn against "bare `bun test`" because of this, meaning the fix is a documentation workaround, not a resolved bug — any dev/CI script that forgets the `--isolate <exact-scoped-dir>/` incantation silently multiplies test execution and risks false confidence in coverage.

(2) **Silent result truncation in sqlite-vec hybrid search.** `src/vector/adapters/sqlite-vec.ts` `query()` over-fetches `limit*3` candidates via `vec0` KNN then filters metadata client-side in JS (lines ~123-133). If the `where` predicate matches fewer than 1-in-3 of the top-K nearest neighbors (plausible for any selective tenant/project filter), the query silently returns fewer than `limit` results with no truncation flag — callers can't distinguish "no more matches exist" from "over-fetch heuristic was too small."

**Recommendation**: (1) Add a CI assertion or wrapper script that fails if `bun test` output includes any path containing `/agents/`; consider excluding `agents/` via `.bunignore` or an explicit roots exclusion instead of relying on developer discipline. (2) In `sqlite-vec.ts`, either iteratively widen the fetch multiplier when filtered results < limit, or surface a `truncated: true` field in `VectorQueryResult`.

## Skeptic
**Finding**: arra-oracle-v3 embeds vanilla `bun:sqlite` (plain WAL, single-writer) and hits exactly the concurrency wall Turso's whole rewrite exists to solve — but never adopts Turso's fix. Evidence is layered across the codebase: `bunfig.toml:7`'s comment admits "parallel test files sharing oracle.db cause 'disk I/O error'" (forcing `isolate=true`); `src/db/index.ts:113-125`'s `isDbLockError()` explicitly string-matches `"disk i/o"`, `"sqlite_busy"`, `"sqlite_locked"`, `"database is locked"`; `src/middleware/errors.ts:114-116` maps those to HTTP 503; and `handleReflect()` (`src/server/handlers.ts:384-396`) papers over a lock hit with a user-facing "Oracle is indexing — please wait..." message instead of resolving the contention. Turso's founding architectural bet was precisely this problem — bolting MVCC onto WAL so concurrent writers don't serialize (per prior research: ~4x throughput under 8-thread compute load vs. plain SQLite). arra-oracle-v3 embeds SQLite the same way Turso does, faces the identical single-writer symptom class, and instead of solving it at the engine level just detects-and-degrades (503 + friendly copy) while relying on test-only isolation to hide it in CI. That's resilience theater around an unsolved problem, not a solved one — and it will surface for real under this product's actual workload (multiple MCP tool calls / plugins / sidecar writers hitting one `oracle.db` concurrently), which is a more concurrent write pattern than most single-user SQLite apps.

**Recommendation**: Either swap the storage engine to libSQL/Turso to inherit its MVCC writer model, or build an internal write-serialization queue with retry/backoff (mirroring the backoff already implemented in `src/vector/fallback-chain.ts`) so `SQLITE_BUSY`/lock errors never reach an HTTP 503 in the first place.

## Architect
**Finding**: The comparison table in the arra-oracle-v3 research is explicit: arra-oracle-v3 is "architecturally closest to Turso's premise" (embedded SQLite as the unit of storage, `src/storage/drizzle-sqlite.ts:74-105`) but is missing "Turso's defining feature — the primary/embedded-replica sync protocol." Today, cross-Oracle data movement (Nexus → Odin → Pulse → Neo, per the sibling-collaboration table in this repo's CLAUDE.md) goes through `src/federation/capability-provider.ts`, which is only a peer/capability *registry* (who exists, what they can do) — there is no mechanism to actually sync `oracle_documents`/`oracle_fts` rows between two Oracles' independent SQLite files. Each of the 191+ Oracles is a fully isolated single-writer file with no replication path.

Turso just solved exactly this problem with its new `@tursodatabase/sync` push/pull protocol, benchmarked (vendor-published, but with open reproducible code at `tursodatabase/turso-sync-benchmark`) at 8.9x faster bulk sync and 312x faster read-your-writes than the old embedded-replica approach, using delta push/pull rather than full-file transfer.

**Recommendation**: Add a `sync` route/tool to arra-oracle-v3 (alongside the existing `/api/federation/*` mesh) that implements a Turso-Sync-style delta push/pull between sibling Oracles' `oracle_documents`/`oracle_fts` tables — using each row's existing bitemporal `valid_time`/`superseded_*` columns as the change-cursor — so "Nothing is Deleted" signal-sharing between Nexus and Odin/Pulse/Neo becomes native DB replication instead of ad-hoc HTTP relay through `FederationCapabilityProvider`.

## Auditor
**Finding**: arra-oracle-v3 has no analogue to the compliance/security-disclosure maturity Turso and Spice.ai both reached at seed stage. Both competitors hold **SOC 2 Type II** attestations (Turso: Jul 2024, pen-tested by Doyensec; Spice.ai: Feb 2024, audited by Prescient Assurance) and publish security-advisory histories via GitHub Security Advisories/OSV (Turso: CVE-2026-48790, CVE-2025-47736; Spice.ai: zero, but confirmed via `gh api .../security-advisories`). arra-oracle-v3 has nothing comparable in the research corpus — no trust portal, no named pen-test vendor, no advisory feed — and its own plugin isolation is explicitly *not* a security boundary: `src/plugins/sandbox.ts`'s docstring states isolation is "error containment only... not process isolation, VM boundaries, or any security boundary" (delegates to `error-containment.ts`). That's a written admission of a gap the competitors have externally audited away.

Separately, arra-oracle-v3's own CI has a documented, unresolved reliability bug: bare `bun test` recurses into `agents/*` git-worktree copies (5 nested `src/`+`tests/` trees), forcing every scoped-test instruction in `AGENTS.md`/`CLAUDE.md` to warn "NOT bare `bun test`" — and ~1000 of ~1025 test files aren't run by any GitHub Actions workflow (`ci.yml`/`test.yml` only exercise a fixed small file list). Turso and Spice.ai, despite their own volatility (Turso's 7x COUNT regression, Spice's OOM bugs), at least run in public CI with reproducible benchmark repos.

**Recommendation**: Before any external security claim, (1) fix `bunfig.toml` root-glob to exclude `agents/` worktrees or gate CI on the full `tests/` tree, and (2) either commission a real pen-test/SOC2 scoping exercise or explicitly document the plugin trust model as "no sandbox" in user-facing docs, not just an internal source comment.

## Market Positioner
**Finding**: These three occupy non-overlapping category boundaries, not a competitive triangle. Turso is a **storage engine vendor** (from-scratch SQLite rewrite, seed-stage ~$7M/~20 employees, no committed 1.0 GA — milestone "1.0" open since Dec 2023 with 148 open issues). Spice.ai is a **federation/acceleration platform** (76-crate DataFusion workspace federating 34 connector types, ~$14.5M/~17-18 employees) — and it *consumes* Turso as one of six pluggable accelerators (1,811 references to "turso" in its own codebase). arra-oracle-v3 is neither: it's an **application-layer memory product** that embeds `bun:sqlite` (not Turso/libSQL) plus hand-rolled FTS5 + pluggable vector adapters (LanceDB default, sqlite-vec, Qdrant), with hybrid ranking (`combineSearchResults`, 0.7/0.3 weights) done entirely in TypeScript app code, never in the storage engine. Architecturally it shares Turso's *premise* ("real embedded SQLite is the unit of storage") but lacks Turso's defining feature — no primary/replica sync protocol, no embedded-replicas; every arra-oracle deployment is just an isolated file with `tenant_id` filtering. It shares zero DNA with Spice.ai's premise (federating *foreign* Postgres/Snowflake/Iceberg via a shared planner) — arra's "federation" (`src/federation/capability-provider.ts`) is a peer-discovery registry between sibling Oracle *processes*, not cross-engine SQL pushdown.

**Recommendation**: Don't position arra-oracle-v3 against Turso or Spice.ai in any GTM material — they aren't in the buyer's consideration set (one sells an OLTP engine, one sells an OLAP federation gateway, arra-oracle sells a finished MCP-native memory product). The only real adjacency worth tracking: if Turso's Turso Cloud (private beta, BYOC/sync) matures, arra-oracle could someday swap its bun:sqlite storage layer for Turso as a *dependency* — mirroring exactly what Spice.ai already did — not compete with it.

## Security Auditor
**Finding**: Turso and Spice.ai both have externally-verified security posture: named SOC 2 Type II attestations with a real auditor (Turso: Doyensec pen-test, "zero issues", 2024; Spice.ai: Prescient Assurance, Feb 2024), plus queryable GitHub Security Advisories showing real disclosed/patched CVEs (Turso: CVE-2026-48790 — turso-cli JWT persisted at world-readable 0o644, CVSS 5.5; CVE-2025-47736 in libsql-sqlite3-parser). Nothing comparable turned up anywhere in the arra-oracle-v3 research — no SOC2/pen-test claim, no CVE/advisory history, because it's a private BUSL-1.1 repo with no public bug-bounty or third-party audit surface. Absence of CVEs here is not evidence of safety, just absence of scrutiny (unlike Turso/Spice, nobody outside the team is looking). Meanwhile the codebase self-documents a concrete gap: `src/plugins/sandbox.ts` is an explicitly deprecated shim whose own docstring says plugin isolation is "error containment only," not "process isolation, VM boundaries, or any security boundary." The unified plugin system (`src/plugins/unified-manifest.ts`) lets any `plugin.json` register MCP tools, API routes, and spawn sidecar server processes — analogous surface to the exact class of bug Turso just patched (credential/token handling by a semi-trusted component), but with no independent pen-test to catch it. The embedder-identity registry (`src/vector/embedder-identity.ts:119-124`) also writes plaintext JSON via tmp+rename with no documented file-permission hardening — the same category of issue as CVE-2026-48790.

**Recommendation**: Before treating arra-oracle-v3 as production-ready for multi-tenant/agent-facing use, commission an external pen-test (even a lightweight one, modeled on Doyensec's turso-cli engagement) specifically targeting the plugin sidecar/token-storage surface, and file permission-audit all JSON registries/credential files written under `~/.oracle`/`~/.arra`.

## Performance Engineer
**Finding**: arra-oracle-v3's hybrid search does over-fetch + in-process JS filtering, not pushdown — sqlite-vec's `query()` fetches `limit*3` candidates when a `where` metadata filter is present, then does `JSON.parse(row.metadata)` + `Object.entries(where).every(...)` in JS (`src/vector/adapters/sqlite-vec.ts:123-133`). This is architecturally the same class of problem the SQG benchmark caught in Turso: a "missing fast-path" pattern where a filter that should be pushed into the engine is instead paid for as 3x over-fetch + per-row JSON parse. At scale this degrades non-linearly exactly like Turso's COUNT case did (5,593 ops/s vs 1.1M for better-sqlite3). Separately, `combineSearchResults()` (`src/server/handlers.ts:306-345`) runs FTS5 and vector legs as two round-trips fused with hand-tuned weights (0.7/0.3, 0.65/0.35, +0.12 overlap) in TS — every hybrid query pays double query-planning overhead that Turso/Spice avoid by pushing fusion closer to the engine (DataFusion) or not needing it (single SQLite planner). Also relevant: `bunfig.toml:7` documents "parallel test files sharing oracle.db cause 'disk I/O error'" — a live symptom of SQLite's single-writer/WAL contention, the exact bottleneck Turso's MVCC rewrite targets and that arra-oracle-v3 has not addressed (no MVCC, `PRAGMA journal_mode = WAL` only, single `bun:sqlite` handle per process).

**Recommendation**: Before any Turso/Spice comparison ships to Nat, run the SQG-style methodology directly against arra-oracle-v3: benchmark `sqlite-vec` query() with and without metadata `where` filters at 100K/1M row scale to quantify the over-fetch tax, and benchmark concurrent-write throughput under the same "8 threads + simulated compute" harness Turso published (`turso.tech/blog/beyond-the-single-writer-limitation`) to get a real single-writer-contention number instead of inferring one from the `bunfig.toml` comment.

## Integration Specialist
**Finding**: All three converge on MCP as the integration layer, but arra-oracle-v3's design already avoids Spice.ai's shipped bug — and also reproduces its risk surface one layer down. Spice.ai's MCP server used "/" in tool names, violating Claude/Cursor naming constraints. arra-oracle-v3's core tool registry (`src/tools/mcp-manifest.ts`) uses flat snake_case names (`oracle_search`, `oracle_learn`, `oracle_trace`, etc.) — structurally immune to that specific defect. But the unified plugin system (`src/plugins/unified-manifest.ts`, `src/mcp/plugin-tools.ts::pluginMcpToolsFrom`) lets third-party `plugin.json` manifests contribute arbitrary `mcpTools[]` entries merged into the same registry (`src/mcp/server.ts:171-193`) — no validation of tool-name characters was found in the research. That's exactly the vector where a plugin author could reintroduce Spice's bug (e.g., naming a tool after a route path like `oracle/search`). Turso sidesteps this entirely by keeping MCP a fixed 9-tool surface baked into the `tursodb` binary, not an extensible registry — safer but less composable, mirroring arra's own `catalog/arra-oracle.yaml` (9 tools declared) already drifting behind the live 24-28-tool manifest, an unmaintained-metadata pattern the research flagged in Spice's own ROADMAP.md-vs-milestones drift.

**Recommendation**: Add a name-validation guard in `normalizeUnifiedPluginManifest` (or at merge time in `pluginMcpToolsFrom`) rejecting any MCP tool name that isn't `^[a-zA-Z0-9_-]+$` — cite Spice.ai's `/`-in-tool-name incident directly in the code comment as the regression this prevents. Also add a CI check (like `tests/docs/readme-claims.test.ts`) asserting `catalog/arra-oracle.yaml`'s tool count matches the live manifest count.

## Minimalist
**Finding**: Spice.ai's `docs/ROADMAP.md` (dated versions v2.1 Jul→v2.5 Jan'27, each linked to a GitHub milestone with named features and a DataFusion version pin) is a single markdown file that gives users and contributors an at-a-glance, falsifiable answer to "what's coming and when." Turso, by contrast, has no such file — its roadmap is scattered across blog posts, an undated README bullet ("vector indexing... on the roadmap"), and GitHub milestones whose content visibly drifts from what any blog says (confirmed in the research: v2.1/v2.2/v2.3 milestone *content* doesn't match ROADMAP.md's own version labels either — but at least the doc exists as a reconciliation point).

arra-oracle-v3 has neither: no `docs/ROADMAP.md` was surfaced anywhere in the three deep code-exploration passes (`docs/` was inventoried as "60+ docs, including `docs/architecture.md`" with zero mention of a roadmap doc). Direction currently lives implicitly in CLAUDE.md's "Short Codes" and scattered `AGENTS.md` notes — nothing dated, nothing milestone-linked, nothing a user/contributor can point at to ask "is X planned."

**Recommendation**: Add one file, `docs/ROADMAP.md`, copying Spice.ai's exact format: a table of upcoming versions with target month, theme, named features, and a link to the corresponding GitHub milestone (arra already tracks work via issues/milestones per the CI/test docs). Zero code, zero risk, ~30 minutes, immediately upgrades external trust and internal alignment — the cheapest possible import of Spice.ai's practice.
